# Cantilune 通俗读本（无需数学背景）

> **关于严谨性的一句话。** 本读本刻意少用符号。但“好读”不等于“对真相含糊”。这里的每一句都有 spec 中的精确版本：静态呈现与恒等视图是定义性的，四条运行时投影则都须有完整证书。现有有限见证尚未证明整个框架；在中央证明和独立复核完成前，它仍是**未经验证**的。

## 1. 大家都有的那种感觉

如果你用过当下的 agent 工具，你多半感受过 cantilune 项目起步时的同一种感觉：它们能用，但只是某种程度上的能用；一旦不行，你很难说清*为什么*。一次运行卡住了。某一步"太早"发生了。两个 agent 似乎对已决定的事各执一词。你重放一次运行，重放显示的和记忆里的微妙不同。这些都不是通常意义上的 bug，它们是一个更深问题的症状。

那个更深的问题是：在这些工具里，**工作的"形状"和工作的"执行"是用两套谁也不连上的语言描述的。** 形状是你画的图、或你写的配置。执行是某人那周写的调度器代码。这两样东西*本应*描述同一件事，但没有任何东西强迫它们一致。于是随时间漂移——图说一套、代码做一套，"实际发生了什么"成了见仁见智。你想要的每一个保证（"不会死锁""能重放""能审计"）都得每次手动、从头论证，去对抗不断变化的代码。

这就是这些工具最终长成那样的原因：有的把形状冻成一块硬模，让什么也漂不动（Cursor 的固定 plan→subagent→loop）；有的放弃形状、让模型即兴发挥，灵活但不可控（Codex）；有的用一个没人完全理解的厚提示/规则层把缝隙撑开（Claude Code）；有的就堆代码，直到自身重量让一切停滞（OpenClaw 系列）。每一种都是*应对*形状与执行之间那个缺失连接的不同方式。

cantilune 的前提：**别再应对那个缺失的连接了，把它建出来。**

## 2. 改变一切的那个小观察

整个项目就支在这样一个观察上。当你编排工作——任何工作，是不是 agent 都行——你其实只用两种方式组合片段：

- **把两件并排放**，让它们都发生（并行）。
- **把一件的输出接到另一件的输入**，让后者等前者（串行）。

就这样。更花哨的东西——循环、分支、扇出、扇入、重试、交接——都由这两种基本动作搭出来，就像厨房里每道菜都由一小组基本操作搭出来。

有趣的是：这两种动作，"并排放"和"接上"，天生就带着几条大家都认同、无需任何人强制、本就成立的规则：

- 三件事串行做，你把它想成"(A 然后 B) 然后 C"还是"A 然后 (B 然后 C)"，顺序是一样的。（串行*可结合*。）
- 中间什么也不做，什么也不改变。（有一个*空操作*是中性单位。）
- 两件事真并排、互不依赖，你换左右顺序，发生的事不变。（并行独立的工作*可交换*。）

数学家给"恰好有这两种动作、这三条规则的地方"起了个名字：**对称幺单范畴（SMC）**。这名字吓人，想法却很友好。它就是指"一个有两种组合方式——并放与接上——且行为讲理的世界"。那两种动作就是你在 spec 里会看到的 $\otimes$（并放）和 $\circ$（接上）；那三条规则就是公理。

关键认识来了：**编排本身就是这样一个东西。** 我们不是*选择*把 SMC 当工具用于编排，就像你可能选 Redis 做缓存那样。我们是在发现编排*本就*是 SMC——就像数据库*本就*是关系代数，不管你听没听过这个词。$\otimes$ 和 $\circ$ 不是我们强加给问题的符号，它们是"你本来就在做的两件事"的名字。这就是 cantilune 的支点：若编排是 SMC，那它的*执行*就应是*在该 SMC 上一步步走*——而非在它之外另写一堆谈论它的调度器代码。一套语言，不是两套。

## 3. 但形状还不是执行

SMC 本身是个*形状*：它告诉你哪些组合合法。它不告诉你形状怎么*动*。一份食谱描述一道菜，但你还得去炒。

所以 cantilune 加一种让形状动起来的方式：**重写**。你脑中要拿住的形象，是白板上画的一张流程图，而"一步执行"就是"按一条规则，把图的一小块擦掉、换成另一小块"。你有一个写着"计划"的框；一步之后，那个框没了，原处是一个写着"计划完成"的框，带一支箭头通向后续。那次替换——匹配一个小模式、换作另一个——就是一步执行。一整次运行，就是一串这样的替换。

为什么这样，而非朴素的状态机？因为下一节我们需要的东西。我们需要*同一次运行*的四种不同描述，都对"每一步是什么"达成一致。若每种描述各有一套独立状态机，"它们一致"就会变成一场吵不完的架。有了重写，运行是*一串*替换，四种描述只是同一串替换的四种读法。一致成了能证明的事，而非希望。

## 4. 一件事上的四副眼镜

于是我们有一个对象——SMC 加重写——我们要从四个角度看它，因为每个角度回答一个不同的人关心的不同问题：

- **DAG 视角**回答*"什么依赖什么？"*——它展示数据从输出到输入的流动。这是开发者在白板上画的那种视图。
- **Petri 视角**回答*"什么能同时跑、什么在等资源？"*——它把资源（一个 API 配额、一个人的注意力、一个上下文窗口）当作会被占用和释放的东西。
- **π 视角**回答*"谁在跟谁说话？"*——它展示 agent 之间的通道及其上的消息。
- **态射视角**回答*"小件如何合成大件、如何把一件换成等价的另一件？"*——这是复用与重构的视图。

关键主张——也是 cantilune 区别于"四个模块粘一起"之处——在于：这是**一件事的四种读法**，而非四件事拼起来。类比：一个数据库可经关系代数、经事务、经存储布局、经并发来描述，而它们最好都描述同一个数据库，否则就是出问题了。cantilune 的四个视图最好都描述同一次运行。

那个“最好”就是项目存亡所在。每条运行时视图都必须证明：每个源事件
对应一个原生目标事件，目标事件能反射回来，终态不会漂移。态射恒等
情形直接成立；DAG 与 Petri 已有有限参考证书，但一般重构结论仍待证。
通信视图（π）最难，因为 half-π (II) 还需要类型化开进程与 FMS 的
交换桥。这些义务都由 RFC-0002 跟踪，它略带戏剧性地被称为项目的
*生死线*。

### 4.1 三件易误读的事（若"一致"感觉含糊，请读这里）

上面那段把很多意思压进了"一致"一词。值得放慢讲，因为"我好像懂了又好像没懂"正发生在这里。

**其一："一致"不等于"长得一样"。** 四个视图描述同一次运行时，对每一步说**本就该说不同的话**——就像同一场事件，财经版报"股价涨3%"，科技版报"发布会如期举行"。它们不必逐字相同，它们要指向**同一个底层事实**。在 cantilune 里，那个底层事实是**一条重写序列**（§3），每个视图是把这条序列*翻译*进自己的语言。"一致"指：每个视图翻译的是*同一条*序列里的*同一*步 $n$——不是一个视图悄悄在描述第5步、另一个在描述第6步。

**其二："不一致"的对立面是"四个引擎吵不完的架"，不是"四份略有不同的打印件"。** 设想你*没*做 cantilune，而是造了四个独立引擎——DAG 引擎、Petri 引擎、π 引擎、态射引擎——各自自己走。现在你想声称"它们描述同一次运行"。可底下没有共享序列；每个引擎是各自一本书。你只能写胶水代码断言"DAG 引擎做 X 时，Petri 引擎该做 Y"。胶水是代码，有 bug；时间戳差一秒；一旦不一致，**没有更高权威**裁决，因为没有"事实本身"，只有四个引擎各说各话。*那*就是吵不完的架。cantilune 以重写序列为唯一事实、四个视图为它的*翻译*来避开它——故不一致是一个可定位的翻译 bug，不是"哪个引擎对"的哲学争论。一本书的四种译本，不是四个作者各写一本再对账。

**其三：任何视图都不得“生造一步”。** 每条投影都需要源到目标的原生
推导和反射证明。既有范畴论能指导静态 DAG 与 pre-net 构造，却不会
替 Cantilune 自动证明动态规则映射、token 来源、删除策略、重放或终态
分类；真正按定义成立的只有态射恒等视图。π 还多一层工作：有限控制
移动性必须同时有操作性的类型化开进程路线和指称性的 FMS 路线，并
证明二者交换。若某事件只能靠弱多步模拟，这就是必须回到 RFC 的设计
变更，不能藏进翻译细节。

### 4.2 一次具体运行，一步的四种读法

设想一次运行，三个 agent——**Planner、Coder、Reviewer**——一个资源约束：Reviewer 需占用单个*人类审批槽*（同一时刻只允许一个任务占用人类注意力）。运行的重写序列有三步：

1. Planner 产出 TaskPlan。
2. Coder 拿 TaskPlan 产出 Code。
3. Reviewer 拿 Code，占用人类审批槽，产出 Feedback。

现聚焦**第2步（Coder 产出 Code）**，看四个视图如何翻译*同一步*：

- **DAG 视角：** 边 `planner→coder` 早已满足；现在边 `coder→reviewer` 上有 Code 工件，该边契约从"待满足"翻为"已满足"。
- **Petri 视角：** "Code 就绪"库所多一个 token；同时"人类审批槽"库所仍是 0 个 token，故 Reviewer 的变迁**还不能触发**——资源没空。
- **π 视角：** Coder 在通道 `c_coder_to_reviewer` 上发消息 `{Code}`。
- **态射视角：** 态射 `f_code : TaskPlan → Code` 被应用；TaskPlan 被消耗，Code 被产生。

这四句**长得完全不同**——"边满足了""库所有 token 且槽还空着""通道发了消息""态射被应用"。它们*不必*长得一样。它们*一致*，因为四者都在翻译**同一条序列里的第2步**。

对比四独立引擎的噩梦：DAG 引擎日志 `14:03:02 边满足`；Petri 引擎日志 `14:03:03 库所出 token`。这是同一事件吗？任一引擎内部都看不出——你写了胶水去配，胶水可能错，时间戳差一秒，你只能猜。cantilune 里第2步*就是*事实；视图是它的翻译；若某视图不合，那是可定位的翻译 bug，不是关于现实的争论。

再把第2步稍改，以示"为何是 π"：设 Coder 不只产出 Code，还在**运行时开了一条全新通道**问 Planner 一个问题——一个"自由对话"动作，该通道在运行创作时并不存在。

- 在 DAG、Petri、态射视角，这只是"多了一条边/一个变迁/一个态射出现"，步子照常推进——这些视图用的是*现成、已证忠实*的翻译，故在此不会生造。
- 在 π 视角，事实是"运行时创建了一条通道、且其上流过一道消息"。*这*正是一座尚未可信的桥可能捏出伪步骤之处：π 视角可能显示一个"通道创建+消息"，却不对应对象里任何真实重写——桥生造了它。证 π 视图忠实，正是防止此事。而我们只对声明通道（request/accept）对话有把握桥稳；完全自由"运行时随便开通道"可能证不出，届时那份自由延后，且如实说明。

这就是为何唯独 π 视图待证——也为何即便如此，另外三视图也不是免检：它们免检*是因为既有数学已为它们担保*，而非因为这个问题不会出现。

## 5. 这到底买来什么（重点）

若四个视图真的一致，那么几件你通常要当独立产品去造的东西，就因形状之故*免费*得来：

- **trace 即运行。** 既然运行是一串替换，trace *就是*那串。你不是"加日志、祈祷记够了"，运行本身就是日志。
- **重放有显式防漂移义务。** 完整事件先化为不含端点的 recipe，再由声明的
  确定性内核以其源状态为输入执行；验证证明该计算确实恢复记录目标。只把日志中
  已存的答案再读一遍，不算重放。
- **四个视图不能互相撒谎。** 因为它们是同一串替换的读法，你在数据流视图、资源视图、通信视图、复合视图里看到的，可证是同一次运行。别的工具里，每个视图是一个独立故事，合不合得上靠运气。
- **"模型决定什么"有一条清晰的线。** 模型决定*一个框内部发生什么*（读输入、产输出、调工具）。框*之间*的一切——什么并行、什么等、谁给谁发消息——由形状和策略决定，不由模型即兴。这就是对"厚控制面"问题的回答：不是更薄的控制面，而是一条你真能指得出的线。

这些不是承诺，它们是形状的*后果*，前提是 §4 的一致成立。那个"前提"正是项目如此组织的原因：先证一致，再在其上建造。

## 6. 下一步去哪

若这些讲得通，你想要精确版本：

- **RFC-0001 §6.1** —— 同样的"编排即 SMC"论证，写给工程师，带恰好够消除歧义的最小符号。
- **`docs/spec/formal-semantics.md`** —— 真正的定义与证明。符号在此，"按构造"之类的主张在此被真正兑现。
- **RFC-0002** —— 专证四视图一致的文档（并诚实报告该证明走到哪了，尤其 π 视图）。

如果你不是数学家，你不必读 spec 就能理解这个项目——你刚读完理解的部分。spec 是给需要*核对*它的人、以及给那几个"感觉对"还不够的决策用的。
